# [백엔드] 인증·토큰 (공통)

> Phase 1 · 로그인 직후 **토큰 발급 → 검증 → 갱신 → 로그아웃** 흐름  
> 공통 규칙: [_공통.md § 인증](../_공통.md#인증)

## 용어 (혼동 방지)

| 용어 | 의미 | 관련 API |
|------|------|----------|
| **토큰 발급** | 로그인 성공 시 Access·Refresh + DB 해시 저장 | `MemberLogin` |
| **토큰 검증** | Access Token·DB 세션 유효성 확인 (전용 API) | `MemberTokenValidate` |
| **토큰 갱신** | Refresh로 새 JWT 쌍 발급 (DB 해시 갱신) | `MemberTokenRefresh` |
| **로그아웃 (JWT Token 비활성화)** | 사용자 로그아웃. DB 해시 NULL. Access만으로 호출 | `MemberTokenRevoke` |

> **권한 = 로그인/관리자**인 API는 토큰 **만료·유효성 검증이 포함**된다.  
> [_공통.md § 권한과 토큰 검증](../_공통.md#권한과-토큰-검증-필수-규칙) · [_공통.md § 토큰 검증](../_공통.md#토큰-검증-공통--로그인-필요-api)  
> `MemberTokenValidate`는 프론트가 **명시적으로** 같은 규칙을 호출하는 전용 API이다.  
> 관리자 회원 정지(`AdminMemberDeactivate`)는 [1.16-AdminMemberDeactivate.md](./1.16-AdminMemberDeactivate.md) — **본 흐름과 별개**.

## API 흐름

```
MemberLogin (1.3)          → 토큰 발급, DB 해시 저장
       ↓
MemberTokenValidate (1.4)  → 토큰 검증 (앱 기동·화면 진입 등)
       ↓
MemberTokenRefresh (1.5)   → 토큰 갱신
       ↓
MemberTokenRevoke (1.6)    → 로그아웃 (JWT Token 비활성화)
```

**중복 로그인:** 새 로그인 시 기존 세션 **JWT 비활성화** → 이전 기기 `1006`/`1005`.
