# [백엔드] 공통 — 응답·에러·공유 타입

## 필드 설명 작성 규칙 (누락)

요청·응답(및 Query/Path) **표의 설명 칸은 그 칸만 보고 의미가 읽혀야** 한다.

- `_공통.md`에 동일 필드가 있어도 **상세 API 문서 표에 의미를 다시 적는다**. 공통은 표준·환산·검증용이다.
- 다음만 적는 것은 **누락**이다: `요청 에코`, `요청값 에코`, `동일`, `위와 같음`, `공통 참고`, `생략`, 빈 칸.
- 링크(`[_공통 § …]`)는 **보조**일 뿐, 설명 칸을 대체하지 않는다.

```markdown
<!-- ❌ 누락 -->
| `offset` | number | 요청 에코 |

<!-- ✅ -->
| `offset` | number | 이번 조회에서 건너뛴 건수. 요청 offset와 동일(보정 시 보정값). 현재 페이지 환산에 사용 |
```

## 응답 래퍼

모든 API는 HTTP 200(또는 파일 다운로드 예외)과 함께 아래 JSON 형태를 사용한다.

### 성공

```json
{
  "resCode": 200,
  "data": { }
}
```

### 실패 (비즈니스·검증)

```json
{
  "resCode": 2002,
  "data": {
    "message": "이미 사용 중인 아이디입니다"
  }
}
```

- `resCode`: 성공은 **200**. 실패(비즈니스·검증)는 **4자리** 정수 에러코드.
- `data`: 성공 시 페이로드 객체. 실패 시 최소 `{ "message": "..." }`.

### 서버 오류

HTTP `500`, `resCode: 9000`

## 인증

- **JWT(Access Token) + Refresh Token** 기반이지만, **서버(DB)가 현재 유효한 세션·토큰을 관리**한다.
- 로그인 성공 시 `accessToken`·`refreshToken` 발급 + DB에 **현재 세션**으로 저장.
- API 호출 시: JWT 서명·만료 확인 후 **DB에 저장된 토큰 해시·만료 시각과 대조**한다.
- **회원당 활성 세션 1개** — 새 로그인 시 기존 세션·토큰 전부 폐기 (**중복 로그인 방지**).
- Access Token 만료 시: `MemberTokenRefresh`로 재발급 (재로그인 불필요, DB 갱신).
- 권한: **관리자 `1`**, **일반 `3`**

### 회원 세션 (`tbl_member_session`)

로그인·재발급·로그아웃의 기준 테이블. Refresh Token만 따로 두지 않고 **세션 1건에 Access·Refresh를 함께** 관리한다.

| 컬럼 | 설명 |
|------|------|
| `session_id` | PK (UUID). Access Token JWT 클레임 `sid`에 포함 |
| `member_id` | 회원 FK |
| `access_token_hash` | 현재 Access Token **해시**. **폐기 시 `NULL`(빈값)** |
| `refresh_token_hash` | 현재 Refresh Token **해시**. **폐기 시 `NULL`(빈값)** |
| `access_expires_at` | Access Token 만료 시각 |
| `refresh_expires_at` | Refresh Token 만료 시각 |
| `revoked` | JWT Token 비활성화 여부. 비활성화 시 `true` (해시 NULL과 함께) |
| `created_at` | 세션(로그인) 생성 시각 |
| `last_accessed_at` | 마지막 API 인증 성공 시각 (선택) |

**단일 세션 규칙**

- 회원당 `revoked=false` 인 세션은 **동시에 1개만** 존재한다.
- **로그인 성공 시**: 해당 `member_id`의 기존 활성 세션에 대해 **JWT Token 비활성화**([로그아웃 (JWT Token 비활성화)](#로그아웃-jwt-token-비활성화) 절차) 후, 새 세션·토큰을 발급한다.
- 다른 기기·브라우저에서 이전에 발급된 토큰으로 API 호출 시 → `1006` (다른 곳에서 로그인됨).

### JWT (Access Token)

| 항목 | 규칙 |
|------|------|
| 알고리즘 | **HS256** |
| 클레임 | `sub`(memberId), `role`, `sid`(sessionId), `exp`, `iat` |
| 만료 | **1일** (86400초). DB `access_expires_at`과 **둘 다** 검증 |
| 설정 키 | `moneylog.security.jwt.secret`, `moneylog.security.jwt.access-token-expiration` |
| 환경 변수 | `MONEYLOG_JWT_SECRET`, `MONEYLOG_JWT_ACCESS_EXPIRATION` |

### Refresh Token

| 항목 | 규칙 |
|------|------|
| 형식 | 불투명(opaque) 랜덤 문자열. JWT 아님 |
| 만료 | **7일** (604800초). DB `refresh_expires_at`과 대조 |
| 저장 | `tbl_member_session.refresh_token_hash` (세션과 1:1) |
| Rotation | 재발급 API 성공 시 **새 Access·Refresh 발급**, DB 해시·만료 시각 갱신 |
| 설정 키 | `moneylog.security.jwt.refresh-token-expiration` |
| 환경 변수 | `MONEYLOG_JWT_REFRESH_EXPIRATION` |

### 권한과 토큰 검증 (필수 규칙)

기능명세 메타의 **권한**이 `로그인` 또는 `관리자`인 API는, 본 기능 로직 실행 **전에** Access Token **유효성·만료 검증이 반드시 포함**된다.

| 권한 | Authorization | 검증 |
|------|---------------|------|
| **없음** | 불필요 | 토큰 검증 안 함 |
| **로그인** | `Bearer` 필수 | 아래 [토큰 검증](#토큰-검증-공통--로그인-필요-api) **전체** 수행 (만료·서명·DB 해시·세션) |
| **관리자** | `Bearer` 필수 | 로그인과 **동일 검증** 후 `role=1` 확인 (`1002`) |

- Controller·Service에 검증 코드를 중복 작성하지 않는다. **공통 필터**가 진입 전에 처리한다.
- 검증 실패 시 본 API의 성공 `data`는 반환하지 않는다. (`1001` / `1006` / `1004` / `1002`)
- 프론트가 앱 기동 등에서 **명시적으로** 같은 검증만 호출할 때: `MemberTokenValidate` (`GET /api/v1/auth/validate`) — 규칙은 로그인 필요 API와 **동일**.

### 토큰 검증 (공통 — 로그인 필요 API)

인증 필터(또는 인터셉터)가 Controller 진입 전에 수행한다. Controller마다 중복 구현하지 않는다.  
**권한 = 로그인/관리자인 모든 API에 포함**된다. (위 [권한과 토큰 검증](#권한과-토큰-검증-필수-규칙))

프론트가 **명시적으로** 같은 규칙을 호출할 때는 전용 API `MemberTokenValidate` (`GET /api/v1/auth/validate`)를 사용한다.

**Access Token (`Authorization: Bearer`) 검증 순서**

```
1. Bearer 토큰 추출. 없으면 1001
2. JWT 서명·형식 검증. 실패 시 1001
3. JWT exp 만료 확인. 만료 시 1001
4. JWT에서 sub(memberId), sid(sessionId) 추출
5. DB: session_id=sid AND member_id=sub 조회. 없으면 1006
6. access_token_hash가 NULL·빈값이면 1006 (JWT Token 비활성화됨)
7. hash(요청 accessToken) == access_token_hash 비교. 불일치 시 1006
8. now < access_expires_at 확인. 지났으면 1001
9. 회원 active 확인. 비활성이면 1004
10. (선택) last_accessed_at 갱신
→ SecurityContext에 memberId, role, sessionId 설정
```

**Refresh Token 검증 (`MemberTokenRefresh`)**

```
1. Body refreshToken 필수. 없으면 9001
2. DB: refresh_token_hash 일치 세션 조회
3. refresh_token_hash가 NULL·빈값이면 1005 (JWT Token 비활성화됨)
4. hash(요청 refreshToken) == refresh_token_hash 비교. 불일치 시 1005
5. now < refresh_expires_at 확인. 지났으면 1005
6. 회원 active 확인. 비활성이면 1004
7. 새 access·refresh 발급, 동일 session_id 행의 해시·만료 시각 갱신 (Rotation)
```

| 구현 | `common-mod` — `JwtTokenProvider`, `MemberSessionService`, `TokenAuthenticationFilter` (가칭) |

**구현 위치 (가이드)**

```
common-mod/.../security/JwtTokenProvider.java
common-mod/.../security/MemberSessionService.java
common-mod/.../security/TokenAuthenticationFilter.java
common-mod/.../config/JwtSecurityProperties.java
data-mod/.../entity/MemberSession.java
data-mod/.../repository/MemberSessionRepository.java
```

**application.yml 예시**

```yaml
moneylog:
  security:
    jwt:
      secret: ${MONEYLOG_JWT_SECRET:change-me-in-production}
      access-token-expiration: ${MONEYLOG_JWT_ACCESS_EXPIRATION:86400}
      refresh-token-expiration: ${MONEYLOG_JWT_REFRESH_EXPIRATION:604800}
```

### 로그아웃 (JWT Token 비활성화)

사용자 **로그아웃** = DB에 저장된 Access·Refresh **해시를 NULL**로 비워 JWT를 더 이상 쓸 수 없게 하는 것.  
**관리자 회원 정지**(`AdminMemberDeactivate`)와 **다른 기능**이다.

요청에 **Access Token만** 사용한다. (Body 없음)

**비활성화 절차** (`MemberTokenRevoke`, 중복 로그인, 비밀번호 변경, 계정 비활성화의 부수 효과 등)

```
1. 대상 tbl_member_session 행 식별
2. access_token_hash  = NULL
3. refresh_token_hash = NULL
4. revoked = true
```

- 이후 Access 검증: 해시 NULL → `1006`
- 이후 Refresh 검증: 해시 NULL → `1005`
- 클라이언트는 로컬 `accessToken`·`refreshToken` 삭제

| API·상황 | 식별 방법 |
|----------|-----------|
| `MemberTokenRevoke` (**로그아웃**) | Header `Authorization` Access Token → `sid` |
| 중복 로그인 | 새 로그인 시 기존 활성 세션 전부 |
| 비밀번호 변경·**관리자 회원 정지** | 해당 `member_id`의 활성 세션에 JWT Token 비활성화 절차 적용 (정지 ≠ 로그아웃 API) |

## 비밀번호 저장 — bcrypt

회원 비밀번호는 **평문 저장 금지**. 서버는 **bcrypt** 로 **단방향 해시**한 뒤 DB에 저장한다. (복호화 불가)  
적용 API: `MemberSignup`, `MemberLogin`, `MemberResetPassword`, `MemberUpdateMe`, `AdminMemberCreate`, `AdminMemberUpdate`.  
(`MemberFindPassword`는 비밀번호를 다루지 않음.)

### 알고리즘

| 항목 | 값 |
|------|-----|
| 방식 | **bcrypt** (단방향 해시) |
| 구현 | Spring Security `BCryptPasswordEncoder` (또는 동등 라이브러리) |
| DB 컬럼 | `tbl_member.pw` — bcrypt 해시 문자열만 저장 |
| 솔트 | bcrypt 해시 문자열 **내장** (별도 컬럼 없음) |

### DB 저장 형식

bcrypt 표준 문자열 (약 60자). 예:

```
$2a$12$LQv3c1yqBWVHxkd0LHAkCOYz6TtxMQJqhN8/X4.G2oQZ2qK5K5K5K
```

| 부분 | 설명 |
|------|------|
| `$2a$` | bcrypt 버전 식별자 |
| `12` | **cost factor** (work factor, 2^12 라운드) |
| 나머지 | 솔트 + 해시 (Base64 인코딩) |

### cost factor — 설정·사용

| 항목 | 규칙 |
|------|------|
| 설정 키 | `moneylog.security.bcrypt-strength` (Spring) |
| 환경 변수 | `MONEYLOG_BCRYPT_STRENGTH` (선택, yml보다 우선) |
| 기본값 | **12** (로컬·운영 공통 기본. 성능에 따라 10~12 조정) |
| 허용 범위 | **10 ~ 14** (그 외 값은 기동 시 검증 실패) |
| 암호화 키 | **없음** (bcrypt는 별도 대칭키 불필요) |
| 사용 모듈 | `common-mod` — `PasswordHasher` (가칭). 백엔드 전용 |

**application.yml 예시**

```yaml
moneylog:
  security:
    bcrypt-strength: ${MONEYLOG_BCRYPT_STRENGTH:12}
```

### 처리 흐름

**저장 (회원가입·비밀번호 변경)**

```
평문 password (HTTPS Body)
  → 규칙 검증
  → PasswordHasher.encode(plain)   # BCryptPasswordEncoder.encode
  → DB tbl_member.pw 에 해시 저장
```

**검증 (로그인)**

```
요청 password (HTTPS Body)
  → DB 에서 member.pw (bcrypt 해시) 조회
  → PasswordHasher.matches(plain, hash)   # BCryptPasswordEncoder.matches
  → false 이면 resCode 1003
```

- API 요청 Body의 `password`는 **항상 평문** (HTTPS 전송). 클라이언트에서 해시하지 않음.
- 로그·AOP에 `password` 필드 **마스킹** (`****`).
- 동일 비밀번호라도 저장마다 해시 문자열이 달라질 수 있음 (솔트 랜덤).

### cost 변경 (이후)

- strength 상향 시 **기존 해시는 그대로 유효**. 로그인 성공 후 선택적으로 `matches` + `encode` 로 재해시(업그레이드) 가능.
- 명세 v1: 가입·변경 시에만 현재 strength 적용.

### 구현 위치 (가이드)

```
common-mod/.../security/PasswordHasher.java
common-mod/.../config/PasswordSecurityProperties.java
```

|------|------|
| 값 예시 | `/api/v1/expend-groups/icons/user01_식비.png` |
| 의미 | `ExpendGroupIconGet` (`GET` 동일 경로) 호출 주소. JSON에 base64를 넣지 않는다 |
| `null` | 아이콘 없음 → 프론트는 플레이스홀더·아이콘 생략 |

### 프론트가 이미지를 받는 방법

아이콘 API도 **로그인 필요** (`Authorization: Bearer <accessToken>`).  
브라우저 `<img src="{iconUrl}">`만 쓰면 Bearer가 안 붙어 **401**이 나는 것이 정상이다. 아래 중 하나로 받는다.

| 방식 | 절차 | 비고 |
|------|------|------|
| **권장 — money-app 프록시** | ① 목록/상세 JSON의 `iconUrl`을 받음<br>② 화면에는 `<img src="/expend-groups/icons/{filename}">` 등 **프론트(앱) URL**을 씀<br>③ money-app이 세션/저장된 토큰으로 백엔드 `ExpendGroupIconGet`을 대신 호출해 바이너리를 중계 | Thymeleaf·서버 렌더에 맞음. `<img>`가 단순해짐 |
| **대안 — fetch + blob** | ① JS에서 `Authorization`을 붙여 `GET {백엔드 origin}{iconUrl}`<br>② `blob` → `URL.createObjectURL` → `<img src>` | SPA·클라이언트 직접 호출 시. 화면 이탈 시 `revokeObjectURL` |

### 요청 수·캐시

- 유형마다 아이콘 GET이 한 번씩 나갈 수 있다. 유형 수가 적을 때(수십 개 이하) **허용**한다.
- 동일 `filename`은 브라우저/`Cache-Control`·프론트 메모리로 **재요청을 줄인다**.
- 목록 JSON에 아이콘 파일을 실지 않는 이유: 응답 비대화·Content-Type 혼선 방지.

관련 API: `ExpendGroupCreate` / `Update`(업로드), `ExpendGroupList`·`Get`·`ListActive`(`iconUrl` 반환), `ExpendGroupIconGet`(바이너리).

## 목록 응답 규칙

여러 건을 반환하는 **목록 API**는 아래 표준을 **반드시** 따른다.

### 표준 구조

```
{ resCode, data: { list: [ { ... }, { ... } ], ... } }
```

| 항목 | 규칙 |
|------|------|
| 필드명 | **`list`** 만 사용 (`items`·`content`·`rows` 등 금지) |
| `list` 타입 | **array** |
| `list` 요소 | **JSON object** 만 허용 (문자열·숫자 등 원시값 배열 금지) |
| 0건 | `"list": []` |
| `data` 루트 | `list`를 **감싸지 않음**. `data` 자체가 배열이면 안 됨 |
| 부가 필드 | 페이징·합계 등은 `list`와 **같은 레벨**. 페이징 시 **`offset`·`limit`·`totalCount` 필수** ([§ 공유 타입 — 페이징](#공유-타입--페이징)). 그 외 합계 등(`expenseTotal` 등)은 도메인명 허용 |

```json
{
  "resCode": 200,
  "data": {
    "list": [
      { "id": 1, "name": "예시" },
      { "id": 2, "name": "예시2" }
    ]
  }
}
```

페이징이 있는 경우:

```json
{
  "resCode": 200,
  "data": {
    "list": [ { } ],
    "offset": 0,
    "limit": 10,
    "totalCount": 42
  }
}
```

요청 Query 예: `?offset=0&limit=10`  
현재 페이지·전체 페이지 수는 응답에 넣지 않는다. 프론트가 `offset`/`limit`/`totalCount`로 계산한다 ([§ 공유 타입 — 페이징](#공유-타입--페이징)).

### 적용 대상 (현재 목록 API — 전부 본 규칙 준수)

| 기능번호 | API 이름 | 페이징 |
|----------|----------|:------:|
| 1.13 | `AdminMemberList` | ✅ |
| 2.2 | `PaymentMethodList` | ❌ |
| 2.6 | `PaymentMethodListActive` | ❌ |
| 2.8 | `ExpendGroupList` | ❌ |
| 2.13 | `ExpendGroupListActive` | ❌ |
| 4.2 | `FixedExpenseList` | ✅ |
| 4.8 | `LedgerMonthlyList` | ❌ (합계 필드 병행) |
| 5.1 | `ExpendTargetList` | ✅ |

신규 목록 API도 동일 표준으로 설계한다.

### 적용 제외

통계 등 **목록 API가 아닌** 응답의 배열(예: `weeklyExpenses[]`, `expendGroupSummaries[]`)은 도메인 의미의 필드명을 유지한다. 다만 요소는 가능하면 **object** 로 둔다.

## 공유 타입 — 페이징

페이징이 **필요한 목록 API**만 적용. 요청·응답에 **같은 축**(`offset`, `limit`)을 두어 프론트가 재사용하고, 서버만 아는 **`totalCount`** 를 추가로 반환한다.

배열 필드는 [§ 목록 응답 규칙](#목록-응답-규칙)의 **`list`** (object 배열).

### 요청 (Query)

| 필드 | 타입 | 필수 | 설명 |
|------|------|:----:|------|
| `offset` | number | ✅ | 건너뛸 건수. **0 이상**, **`limit`의 배수** |
| `limit` | number | ✅ | 가져올 건수. **1 이상** |

### 응답 (`data`)

| 필드 | 타입 | 설명 |
|------|------|------|
| `list` | array (object[]) | 목록. 각 요소는 JSON object |
| `offset` | number | 이번 조회에서 건너뛴 건수. 요청 `offset`과 동일(보정 시 보정값). 프론트가 현재 페이지 환산에 재사용 |
| `limit` | number | 이번 조회에서 가져온 최대 건수. 요청 `limit`과 동일(보정 시 보정값). 프론트가 페이지 크기·전체 페이지 수 환산에 재사용 |
| `totalCount` | number | 조건에 맞는 **전체 건수** (`list` 길이·현재 페이지 건수가 아님) |

### 응답에 넣지 않는 필드

`page`, `pageSize`, `totalPages`, `currentPage`, `size`, `total`, `totalElements` 등. 프론트가 아래 식으로 계산한다.

### 프론트 환산 (재사용)

```text
페이지 크기     = limit
현재 페이지     = (offset / limit) + 1          // 1-based
전체 페이지 수  = ceil(totalCount / limit)      // limit > 0
다음 offset     = offset + limit
```

### 내부 (Data JPA)

```text
요청 offset, limit  →  OFFSET offset LIMIT limit
                    또는 PageRequest.of(offset / limit, limit)  // 0-based page index
totalCount          ←  count 쿼리 / Page.getTotalElements()
```

### 검증

| 조건 | 처리 |
|------|------|
| `offset`·`limit` 누락 | `9001` 등 |
| `limit <= 0` | `9001` 등 |
| `offset < 0` 또는 `offset % limit != 0` | `9001` 등 |
